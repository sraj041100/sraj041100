## `{ featured_projects }` &nbsp;⟶&nbsp; Shipped Products

<details>
<summary><strong>🔷 College Grievance Management System (CGMS)</strong> &nbsp;—&nbsp; <em>Enterprise Workflow Automation</em></summary>

<br/>

> An end-to-end institutional grievance resolution platform built to digitize and streamline the complaint lifecycle across a multi-stakeholder academic environment — eliminating paper trails, reducing resolution latency, and providing real-time administrative visibility.

<br/>

| Attribute | Details |
|:----------|:--------|
| **Stack** | C# · Microsoft SQL Server · .NET Framework · Windows Forms |
| **Architecture** | Multi-tier, role-based access control (RBAC) |
| **Scale** | Institution-wide deployment — Students · Faculty · Administration |
| **Performance** | Automated list generation and dashboard-driven monitoring |
| **Security** | Role-specific module access, authenticated session management |
| **Data Integrity** | Proof document upload, audit trails, structured report generation |
| **Impact** | Eliminated manual grievance tracking; enabled end-to-end digital resolution workflow |
| **Repository** | [![GitHub](https://img.shields.io/badge/View%20Repository-181717?style=flat-square&logo=github&logoColor=white)](https://github.com/sraj04) |

<br/>

**Engineering Scope:**

The system was architected to serve three distinct user classes — students, faculty members, and administrators — each provisioned with a dedicated module and scoped data access. The student interface enables grievance submission with supporting document attachment and live status tracking. Faculty modules allow triage, assignment, and resolution workflows. The administrative console aggregates all grievances into dynamic dashboards with automated classification lists and exportable reports.

The data layer leverages Microsoft SQL Server with normalized schema design, foreign-key integrity, and stored procedures for performance-critical reporting queries. The application enforces strict session-based authentication, ensuring that users can only access data appropriate to their institutional role — a non-trivial security requirement in a multi-department deployment.

</details>
